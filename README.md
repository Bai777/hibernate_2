# Замечания по структуре БД "movie"

В ходе анализа схемы обнаружены следующие проблемные места (ляпы), которые могут нарушать целостность данных:

1. **Таблица `film_text`**
    - Отсутствует внешний ключ (foreign key) на `film.film_id`.
    - Хотя поле `film_id` является первичным ключом таблицы `film_text`, нет явного ограничения, гарантирующего, что значение `film_id` существует в таблице `film`.
    - **Предложение:** добавить `CONSTRAINT fk_film_text_film FOREIGN KEY (film_id) REFERENCES film(film_id) ON DELETE CASCADE`.

2. **Таблица `payment`**
    - Поле `rental_id` допускает NULL (в некоторых версиях Sakila). Однако логически каждый платёж должен быть привязан к конкретной аренде (кроме предоплат?).
    - **Предложение:** сделать `rental_id NOT NULL` и добавить FK на `rental(rental_id)`.

3. **Отсутствие каскадного удаления**
    - При удалении `film` записи в `film_actor` и `film_category` не удаляются автоматически (хотя FK с `ON DELETE CASCADE` присутствуют в последних версиях Sakila, стоит проверить).
    - **Рекомендация:** убедиться, что все связующие таблицы имеют `ON DELETE CASCADE` или использовать триггеры.

4. **Типы данных для дат**
    - В некоторых таблицах используется `TIMESTAMP`, в других `DATETIME`. Для единообразия лучше везде применить `TIMESTAMP` или `DATETIME` с часовым поясом.

5. **Индексы**
    - На полях `last_update` нет индексов, хотя они часто используются в условиях `WHERE` для поиска изменённых записей (например, при репликации).
    - **Предложение:** добавить индекс на `last_update` во все таблицы, где это поле присутствует.
